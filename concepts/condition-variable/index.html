    条件变量（condition_variable）¶

    [!note] 条件变量让线程在"条件不满足"时原子地释放锁并休眠，被唤醒后自动重新获取锁并再次检查条件——这是生产者-消费者、线程池等场景中最核心的同步原语。

    1. 定义与原理¶

    std::condition_variable 解决互斥锁本身无法覆盖的问题：当共享状态不满足要求时，等待线程应该释放锁并休眠，而不是持锁空转（busy-wait）。

    核心操作只有三个： - wait(lock) / wait(lock, predicate) — 原子地释放锁并挂起 - notify_one() — 唤醒一个等待线程 - notify_all() — 唤醒所有等待线程

    为什么必须配合互斥锁？¶

    条件变量本身不保护任何数据。共享条件的检查和修改必须用互斥锁保护，否则必然出现竞态：线程 A 检查条件为 false → 线程 B 修改条件并 notify → 线程 A 挂起（丢失唤醒）。wait() 内部的"原子释放锁 + 挂起"序列是操作系统提供的保证，防止这个窗口期。

    2. 底层机制¶

    wait() 内部的原子序列（以 Linux futex 为例）：

    1. 调用者已持有 unique_lock
    2. wait 将线程放入等待队列（futex wait）
    3. 原子地释放互斥锁
    4. 线程挂起，不再消耗 CPU
    5. 收到 notify 时，OS 将线程移出等待队列
    6. 线程被调度后重新获取互斥锁
    7. 返回给调用者（此时再次持有锁）

    步骤 3-4 之间的原子性至关重要——不会出现"锁已释放但线程还没挂起，通知已发出"的丢失唤醒。

    3. 核心技术细节¶

    虚假唤醒（Spurious Wakeup）¶

    线程可能在没有 notify 的情况下被唤醒——这是 OS 调度层面的行为（信号、调试器断点等），C++ 标准允许但不要求避免。因此条件检查必须放在循环中：

    // 推荐写法：带谓词的 wait，内部展开为 while(!pred()) wait(lock)
cv.wait(lock, [] { return ready; });

// 等价于：
while (!ready) {
    cv.wait(lock);
}

    为什么会有虚假唤醒？——底层原理¶

    虚假唤醒不是 bug，而是操作系统为性能做出的设计取舍。C++ 标准库的条件变量底层依赖操作系统的同步原语，不同平台上触发虚假唤醒的机制不同：

    Linux（futex）：

    condition_variable::wait() 底层调用 futex(FUTEX_WAIT, ...) 系统调用。内核在以下情况会让 FUTEX_WAIT 返回：

    1. 匹配唤醒：另一个线程调用了 FUTEX_WAKE（正常 notify）
    2. 信号中断（EINTR）：进程收到信号（如 SIGCHLD、SIGALRM），内核打断睡眠态。futex 无法区分"真的被 notify 了"和"被信号打断了"，只能返回让用户态重新检查
    3. 竞态窗口：futex 的原子性保证"值检查→入队→挂起"这个序列——但如果有多个等待线程且正在做 FUTEX_CMP_REQUEUE（将等待者从一个 futex 迁移到另一个），旧 futex 上的线程会被"踢醒"重新排队

    Windows（SRWLOCK / CONDITION_VARIABLE）：

    SleepConditionVariableCS 底层基于 KeWaitForSingleObject 内核调度对象。触发伪唤醒的路径： - APC（Asynchronous Procedure Call）：内核可能在任意时刻向线程投递 APC（如 QueueUserAPC），导致等待提前返回 - 内核调试事件：调试器附加/分离、OutputDebugString 的系统回调会打断等待状态

    POSIX 标准的态度：

    pthread_cond_wait 的规范明确写道"spurious wakeups may occur"——这不是实现缺陷，而是故意留给操作系统的优化空间。如果 POSIX 要求"绝不虚假唤醒"，那么每个 pthread_cond_signal 都要在内核中做精确的排队和计数——对于高并发的高频通知场景，这些开销是灾难性的。

    结论：虚假唤醒的本质是内核为了并发性能，用"可能多唤醒一次"换取"不用在每个调用上做昂贵的精确计数"。用户态用 while(!pred()) 循环做最终裁决——成本极低（只是一次轻量级检查），换来内核实现的简洁和高效。

    丢失唤醒（Lost Wakeup）¶

    notify 发生在 wait 之前的经典 bug。用带谓词的 wait 可避免——即使 notify 已发出，谓词检查会直接通过而不挂起。

    notify 在锁内 vs 锁外¶

    • 锁外通知（推荐）：被唤醒的线程可以直接获取锁，避免 hurry-up-and-wait
    • 锁内通知：保证修改对唤醒者可见，但被唤醒者立即阻塞等待锁释放
    • 不确定时，锁外通知不会引入正确性问题

    notify_one vs notify_all¶

    notify_one notify_all
    唤醒数量 1 个（不确定哪个） 全部
    适用场景 任务队列新增一个元素 全局状态变更、stop 标志
    开销 低 高（但最安全）

    为什么必须用 unique_lock 而不能用 lock_guard？¶

    wait() 内部需要临时释放锁再重新获取。lock_guard 不支持 unlock()/lock() 操作——它是纯 RAII 包装，构造即锁、析构即解。unique_lock 提供可控的锁生命周期。

    4. 经典模式：有界阻塞队列¶

    template<typename T>
class BoundedQueue {
    std::queue<T> queue_;
    size_t capacity_;
    std::mutex mtx_;
    std::condition_variable not_full_;   // 生产者等待
    std::condition_variable not_empty_;  // 消费者等待
    // ...
};

    为什么需要两个条件变量？ 如果生产和消费共用一个 CV： - 消费者取完元素后 notify，可能唤醒另一个消费者（它发现队列空，继续等） - 生产者放入元素后 notify，可能唤醒另一个生产者（它发现队列满，继续等） - 两个 CV 各管一个等待方向，避免错误的唤醒目标

    5. 常见误区¶

    • 用 lock_guard 而非 unique_lock：编译错误——wait 需要可控锁生命周期
    • 不带谓词的 wait 后不循环检查条件：虚假唤醒导致逻辑错误
    • 用条件变量做一次性事件：std::future/std::promise 更简洁
    • 认为 notify_all 让所有线程同时执行：它们仍需逐个获取互斥锁，实际串行
