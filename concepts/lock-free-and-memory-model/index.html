    无锁编程与 C++ 内存模型¶

    [!note] 无锁编程不是"完全不使用同步"，而是用原子操作和内存序替代互斥锁，在高竞争场景下获得更低延迟和更可预测的性能——代价是复杂度急剧上升。

    1. 定义与原理¶

    C++ 内存模型¶

    C++ 内存模型定义了多线程程序中读写的可见性和重排边界。核心概念：

    • 数据竞争（data race）：两个线程同时访问同一内存位置，至少一个是写操作，且没有同步机制介入——导致未定义行为
    • happens-before：A happens-before B 意味着 A 的副作用对 B 可见。这是理解同步的关键术语
    • 内存序（memory order）：控制原子操作之间的重排约束和可见性保证

    六个内存序级别¶

    内存序 约束强度 典型场景
    memory_order_seq_cst 最强（全局一致顺序） 默认选项，最易理解
    memory_order_acquire 读侧：后续读写不能重排到此之前 与 release 配对
    memory_order_release 写侧：之前读写不能重排到此之后 与 acquire 配对
    memory_order_acq_rel 兼具 acquire + release read-modify-write 操作
    memory_order_consume 比 acquire 弱，仅依赖序 极少使用，多数编译器退化为 acquire
    memory_order_relaxed 最弱（仅保证原子性，无同步） 计数器递增等无需可见性保证的场景

    发布-获取（Release-Acquire）配对¶

    // 线程 A（发布）
data = 42;
ready.store(true, std::memory_order_release);  // 在此之前的所有写入对 B 可见

// 线程 B（获取）
if (ready.load(std::memory_order_acquire)) {    // 在此之后的所有读取保证看到 A 的写入
    use(data);  // 保证看到 42
}

    无锁编程（Lock-Free）¶

    • lock-free：系统整体至少有一个线程能持续推进（即使其他线程被挂起）
    • wait-free：每个线程都能在有限步内完成——最强保证，实现难度最高

    无锁编程依赖原子操作（特别是 CAS），通过重试循环而非阻塞等待来实现同步。

    2. SPSC 环形队列详解¶

    SPSC（Single Producer, Single Consumer）是无锁编程的入门级结构——通过约束"单生产者 + 单消费者"将同步复杂度控制在可管理范围：

    template<typename T, size_t N>
class SpscQueue {
    std::array<T, N> buffer_;
    std::atomic<size_t> head_{0};  // 消费者独写
    std::atomic<size_t> tail_{0};  // 生产者独写

    bool push(const T& value) {
        auto tail = tail_.load(std::memory_order_relaxed);  // 只有生产者会写 tail
        auto next = (tail + 1) % N;
        if (next == head_.load(std::memory_order_acquire))  // 获取消费者的最新 head
            return false;  // 队列满
        buffer_[tail] = value;
        tail_.store(next, std::memory_order_release);  // 发布数据给消费者
        return true;
    }

    std::optional<T> pop() {
        auto head = head_.load(std::memory_order_relaxed);
        if (head == tail_.load(std::memory_order_acquire))  // 获取生产者的最新 tail
            return std::nullopt;  // 队列空
        T value = buffer_[head];
        head_.store((head + 1) % N, std::memory_order_release);
        return value;
    }
};

    内存序选择原理： - push 读 head_ 用 acquire：需要看到消费者对 head_ 的更新后的 buffer 状态 - push 写 tail_ 用 release：确保 buffer 写入在 tail 更新前完成 - push/pop 读自己的索引用 relaxed：单线程独写，无竞争

    3. 无锁编程的主要问题¶

    ABA 问题¶

    值从 A→B→A，CAS 看到值仍是 A 就以为状态没变——但实际上中间已经发生了 B。解决方法：加版本号（如 std::atomic<std::pair<T*, uint64_t>>）或用 tagged pointer。

    伪共享（False Sharing）¶

    两个线程操作不同的原子变量，但它们在同一缓存行（cache line，通常 64 字节）内——每次写入导致对方的缓存行失效。解决：用 alignas(64) 或填充 padding 将变量隔离到不同缓存行。

    活锁¶

    线程不断重试 CAS 但每次都失败——虽然系统在前进（lock-free），但某些线程可能长时间无法完成。解决：加退避策略（exponential backoff）。

    4. 工程决策：锁 vs 无锁？¶

    互斥锁 无锁
    正确性 容易推理 极难验证
    调试 死锁可检测 难以复现
    延迟 阻塞等待时有上下文切换开销 稳定低延迟
    吞吐 低竞争下接近无锁 高竞争下优势明显
    开发成本 低 极高

    建议路径：先用锁实现正确版本 → profiling 确认瓶颈 → 从 SPSC 等约束结构入手替换 → 严格控制无锁代码的边界。

    5. 关联概念¶

    • condition-variable — 锁同步的另一极，适合需要等待条件的场景
    • raii — lock_guard/unique_lock 是锁同步的 RAII 基础
    • lock-guard-vs-unique-lock — 互斥锁的两种 RAII 封装
    • [[atomic-vs-volatile]] — std::atomic 和 volatile 常被混淆，但语义完全不同

    6. 面试要点¶

    • happens-before 是核心术语——用它解释同步关系的建立
    • 内存序不是"性能开关"——乱用 relaxed 是常见错误，默认 seq_cst 是安全的起点
    • ABA 问题要能举例说明（lock-free stack 的经典案例）
    • 伪共享要能解释缓存一致性协议层面的原因（MESI）
