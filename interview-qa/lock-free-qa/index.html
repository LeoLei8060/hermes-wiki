    无锁编程与内存序面试题集¶

    Q1：C++ 内存模型解决什么问题？happens-before 是什么？¶

    难度：⭐⭐⭐

    涉及知识点：lock-free-and-memory-model

    掌握程度：未掌握

    问题¶

    C++ 内存模型是什么？为什么要引入这个东西？"happens-before"关系怎么理解？

    回答要点¶

    C++ 内存模型解决的是多线程环境下读写操作的可见性和重排问题。

    问题来源： - 编译器可能重排指令（优化） - CPU 可能乱序执行和缓存延迟 - 线程 A 写 x=1; y=2，线程 B 可能先看到 y=2 再看到 x=1

    happens-before：A happens-before B 意味着 A 的所有副作用对 B 可见。这是多线程同步的语义基础。

    建立 happens-before 关系的方式： - 同一个线程内：前面的语句 happens-before 后面的语句（程序顺序） - mutex::unlock() happens-before 后续的 mutex::lock()（锁同步） - atomic::store(release) happens-before 对应的 atomic::load(acquire)（原子同步）

    常见追问¶

    • happens-before 和 sequenced-before 的区别？（前者是多线程间的，后者是单线程内的求值顺序）

    Q2：六个 memory_order 分别适用什么场景？¶

    难度：⭐⭐⭐

    涉及知识点：lock-free-and-memory-model

    掌握程度：未掌握

    问题¶

    从 relaxed 到 seq_cst，不同内存序的选择标准是什么？能不能给每个级别举一个实际场景？

    回答要点¶

    内存序 场景 为什么？
    relaxed 计数器递增 counter++ 只关心最终数值，不关心与其他写入的顺序
    无需同步保证
    acquire 消费者读 SPSC 队列的 head_ 需要看到生产者对 buffer 的写入
    release 生产者写 SPSC 队列的 tail_ 确保对 buffer 的写入先于 tail_ 的更新
    acq_rel CAS 循环中的 read-modify-write 同一个操作既需要读到最新值又要发布给他人
    seq_cst 并发标志位的简单同步 全局一致顺序，推理最简单

    默认选择：不确定时用 seq_cst，正确性优先。只在确认性能瓶颈时才降级。

    常见追问¶

    • acquire 和 consume 的区别？（consume 更弱的依赖序，实际编译器中几乎都退化为 acquire）

    Q3：lock-free 和 wait-free 的区别？ABA 问题是什么？¶

    难度：⭐⭐⭐

    涉及知识点：lock-free-and-memory-model

    掌握程度：未掌握

    问题¶

    无锁编程中有 lock-free 和 wait-free 两个概念，区别在哪？ABA 问题怎么解决？

    回答要点¶

    lock-free：系统整体保证至少有一个线程能持续推进。如果某个线程被挂起，其他线程不受影响。但某个特定线程可能重试无数次（活锁）。

    wait-free：每个线程都能在有限步内完成——更强的保证，实现难度极高。

    ABA 问题： 

    初始：栈顶 = A
线程1：读栈顶 = A，准备 CAS 到 B
线程2：pop A → pop B → push A  // 值回到 A，但中间已发生状态变化
线程1：CAS 成功（以为值没变） → 实际已丢失线程2 的修改
    解决：加版本号（tagged pointer）、hazard pointer 或 epoch-based reclamation。

    常见追问¶

    • 互斥锁算什么级别？（阻塞同步，既不是 lock-free 也不是 wait-free——一个线程持有锁时其他线程完全无法推进）
