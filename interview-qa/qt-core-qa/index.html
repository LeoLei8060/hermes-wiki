    Qt 核心机制面试题集¶

    Q1：信号槽跨线程时，槽函数在哪个线程执行？¶

    难度：⭐⭐⭐

    涉及知识点：qt-event-system、moc

    掌握程度：未掌握（第一轮面试答反）

    问题¶

    信号在子线程发射，连接到一个主线程的 widget 槽函数。这个槽到底在哪个线程执行？为什么？

    回答要点¶

    答案：在主线程执行——前提是使用默认的 AutoConnection。

    原因链条： 1. AutoConnection 在 connect 时比较信号发送者和槽接收者的线程亲和性 2. 二者线程不同 → 自动退化为 QueuedConnection 3. QueuedConnection 将槽调用封装为 QMetaCallEvent，投递到接收者线程的事件队列 4. 接收者线程的事件循环取出事件，在接收者线程中执行槽

    不是：信号在哪个线程发射，槽就在哪个线程执行——这是 DirectConnection 的行为。

    常见追问¶

    • 如果我用 Qt::DirectConnection 强制跨线程直接调用呢？（槽在信号线程执行，widget 操作 → 崩溃或未定义行为）
    • 子线程没有事件循环会怎样？（排队槽永不执行——事件积压在队列里）

    Q2：postEvent 和 sendEvent 的区别？¶

    难度：⭐⭐

    涉及知识点：qt-event-system

    掌握程度：未掌握（第一轮面试答反）

    问题¶

    QCoreApplication::postEvent() 和 sendEvent() 的区别？各自适合什么场景？

    回答要点¶

    postEvent sendEvent
    执行方式 异步——加入事件队列后立即返回 同步——立即调用接收者的 event()
    调用栈 不进入当前调用栈 在当前调用栈内直接执行
    线程安全 线程安全——可跨线程投递 仅限接收者所在线程调用
    事件所有权 由事件循环接管，处理后自动 delete 调用者仍拥有事件对象
    返回值 不返回处理结果 返回 event() 的 bool

    面试加分：postEvent 的"线程安全"是因为 Qt 内部使用互斥锁保护事件队列——这是它与裸指针操作的关键区别。

    Q3：MOC 为什么不能用纯 C++ 模板替代？¶

    难度：⭐⭐⭐

    涉及知识点：moc、qt-event-system

    掌握程度：部分掌握

    问题¶

    Qt 的 MOC 常被诟病为"非标准 C++"。为什么 Qt 不用模板来实现信号槽？其他库（如 boost::signals2）不是做到了吗？

    回答要点¶

    模板能做到信号槽的编译期连接，但做不到 Qt 需要的运行时能力：

    1. QMetaObject::invokeMethod(name)：按字符串调用槽——需要运行时函数名到函数指针的映射。模板在编译期展开后，运行时不保留这些信息。
    2. 属性系统：Q_PROPERTY 需要按名称读写属性——模板做不到运行时字符串到 getter/setter 的映射。
    3. qobject_cast：跨动态库边界的运行时类型转换——typeid 依赖 RTTI 且在动态库边界不可靠。
    4. QueuedConnection：把槽调用"序列化"为事件，延迟到目标线程事件循环中执行——需要把任意参数类型的调用封装成统一的 QMetaCallEvent。纯模板需要为每种参数组合生成不同的实例化，无法统一投递。

    boost::signals2 提供信号槽功能但没有属性系统、按字符串调用、跨线程排队的能力。Qt 选 MOC 是因为它需要的是完整的元对象系统而不只是信号槽。

    常见追问¶

    • C++20 的 std::source_location + C++26 的反射提案能让 Qt 抛弃 MOC 吗？（理想情况下可以，但需要编译器生态完全跟上）
