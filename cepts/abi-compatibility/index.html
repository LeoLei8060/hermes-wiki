    ABI 与二进制兼容性¶

    [!note] ABI 不是"理论概念"，而是动态库升级时旧二进制能不能继续跑的硬约束。SDK 发布、插件系统、热更新都依赖对 ABI 破坏条件的准确判断。

    1. API vs ABI¶

    API ABI
    层面 源码级接口 编译后二进制级约定
    兼容性检验 重新编译通过 不重新编译，旧二进制能跑
    变化影响 调用方需要改代码 调用方不需要改代码就能崩溃

    通俗说法：API 兼容 = 源码还能编译；ABI 兼容 = 旧 .exe 还能跑在新 .dll/.so 上。

    2. 破坏 ABI 的常见改动¶

    🔴 必然破坏¶

    改动 破坏原因
    增删/重排公开类的数据成员 sizeof 变化，旧代码访问错误偏移
    增删/重排虚函数 虚表布局变化，旧代码调用错误的函数槽位
    修改内联函数的实现 内联代码被编译进调用方，不会自动更新
    修改函数签名（参数类型/返回值） 符号名变化，链接失败
    修改模板默认参数 实例化符号变化

    🟡 可能破坏¶

    改动 条件
    新增非虚成员函数 不影响对象布局，但需要新符号存在
    在类末尾新增数据成员 如果旧代码不会有 sizeof 依赖或 new 该类 → 可能 OK
    修改 enum 值 如果旧代码编译进了具体数值

    🟢 通常安全¶

    改动 条件
    新增非成员函数 不修改已有符号
    新增类（不修改已有类） 独立符号
    Pimpl 类的内部改动 实现藏在 .cpp 中，头文件不变

    3. 工程防御策略¶

    Pimpl（Pointer to Implementation）¶

    // 头文件：公开的稳定接口
class DataEngine {
public:
    DataEngine();
    ~DataEngine();                 // 析构必须在 .cpp 中实现
    DataEngine(DataEngine&&) noexcept;  // 移动构造也要在 .cpp
    DataEngine& operator=(DataEngine&&) noexcept;

    void process(const QString& input);
    QString result() const;

private:
    struct Impl;
    std::unique_ptr<Impl> impl_;   // 实现细节完全隐藏
};

// .cpp：可以随意改
struct DataEngine::Impl {
    QThreadPool pool;
    QCache<QString, Result> cache{1000};
    // 增减成员不影响 ABI —— 头文件不暴露
};

    C 接口封装¶

    // 对外暴露纯 C 接口 —— ABI 最稳定
extern "C" {
    EXPORT void* create_engine();
    EXPORT void  process_data(void* engine, const char* input);
    EXPORT void  destroy_engine(void* engine);
}
// 内部实现可以是任意 C++ 代码

    跨模块边界规则¶

    规则 原因
    不跨边界传递 STL 容器 不同标准库版本的 std::string 布局可能不同
    不跨边界抛异常 异常 ABI 在不同编译器间不兼容
    不跨边界 new / delete 不同模块可能链接不同堆实现
    统一编译器和编译选项 MSVC/Clang/GCC 之间 ABI 互不兼容

    4. Qt 开发中的 ABI 实践¶

    • Qt 主版本间 ABI 不兼容：Qt5 ↔ Qt6 必须重新编译
    • 同主版本、不同次版本：Qt 承诺向前兼容（Qt 6.5 的库能跑 Qt 6.2 编译的程序）
    • Debug/Release 混用：Windows MSVC 下 Debug/Release 的 STL 布局不同 → 崩溃
    • 插件系统的 ABI 要求：插件和主程序必须用同一版本、同一编译器编译（参见 qt-plugin-system）

    5. 关联概念¶

    • qt-plugin-system — 插件系统是 ABI 约束最严苛的场景
    • moc — MOC 生成的代码也受 ABI 约束
    • smart-pointer-comparison — unique_ptr + Pimpl = ABI 稳定的标配
    • qt-deployment — 部署时必须确保所有模块使用兼容的运行时库
