    C++ 编译与链接流程¶

    [!note] C++ 从源码到可执行文件的四阶段流水线——预处理、编译、汇编、链接——每阶段有不同的错误类型和排查策略。理解这个流程是解决 90% 构建问题的基础。

    1. 四阶段流水线¶

    main.cpp
  │
  ▼ ① 预处理 (Preprocessor)
  │  展开 #include、替换 #define、处理条件编译
  │  错误类型: 找不到头文件、宏语法错误
  ▼ main.i（预处理后的翻译单元）
  │
  ▼ ② 编译 (Compiler)
  │  词法分析 → 语法分析 → 语义分析 → 中间代码 → 优化 → 汇编代码
  │  错误类型: 语法错误、类型不匹配、模板实例化失败
  ▼ main.s（汇编代码）
  │
  ▼ ③ 汇编 (Assembler)
  │  汇编代码 → 机器码 → 目标文件（.o / .obj）
  │  错误类型: 极少（除非内联汇编出错）
  ▼ main.o（目标文件：机器码 + 符号表 + 重定位信息）
  │
  ▼ ④ 链接 (Linker)
  │  合并所有 .o + 静态库 → 解析符号引用 → 输出可执行文件
  │  错误类型: undefined reference、multiple definition
  ▼ a.out / main.exe

    2. 一个翻译单元（Translation Unit）¶

    一个 .cpp 文件 + 所有被 #include 的头文件 = 一个翻译单元
    • 每个翻译单元独立编译——编译器看不到其他 .cpp 的内容
    • 这是"为什么模板定义要放头文件"的根本原因
    • 也是"为什么在 a.cpp 定义全局变量，b.cpp 用 extern 声明"的原因

    3. 常见错误的分阶段定位¶

    错误信息 阶段 排查方向
    fatal error: xxx.h: No such file ① 预处理 检查 include 路径（-I）
    error: expected ';' before ② 编译 语法错误
    error: no matching function ② 编译 类型推导失败、重载决议
    error: incomplete type ② 编译 只有前向声明，缺少完整定义
    undefined reference to 'foo' ④ 链接 缺少 .cpp 实现 或 .o/.lib 未链接
    multiple definition of 'foo' ④ 链接 头文件中定义了非 inline 函数/变量
    undefined reference to vtable ④ 链接 Q_OBJECT 头文件未被 MOC 处理

    undefined reference 的破解思路¶

    # 1. 确认符号是否存在
nm -C mylib.a | grep "MyClass::foo"

# 2. 确认链接顺序（依赖项放后面）
g++ main.o -lmylib -lotherlib  # mylib 依赖 otherlib

# 3. C/C++ 混编检查
extern "C" void foo();  // C 函数在 C++ 中必须声明

    4. 静态库 vs 动态库¶

    静态库 (.a / .lib) 动态库 (.so / .dll)
    链接时机 编译时嵌入可执行文件 运行时加载
    可执行文件大小 较大 较小
    更新方式 重新编译 替换 .so/.dll（ABI 兼容前提）
    符号冲突 可能 各库独立空间
    部署 单文件 需附带 .so/.dll

    5. ODR（单一定义规则）¶

    同一个翻译单元内，任何实体只能有一个定义。跨翻译单元时： - 函数/全局变量 → 只能有一处定义，否则 multiple definition - 类/模板/内联函数 → 每个翻译单元可有一份定义，但必须完全相同 - inline 函数允许多处定义，但要求定义完全一致

    6. 关联概念¶

    • abi-compatibility — 链接阶段决定了 ABI 兼容性是否能被满足
    • moc — MOC 是编译流程中的额外预处理步骤
    • cpp-templates — 模板的实例化时机和链接可见性
    • qt-deployment — windeployqt 解决的就是动态库依赖收集问题
