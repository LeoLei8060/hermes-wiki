    Qt 事件系统全景¶

    [!note] Qt 的三大核心机制——事件循环、信号槽、线程亲和性——不是三个独立的概念，而是一个统一的事件驱动模型的三层视图。理解它们的交叉点才是面试关键。

    1. 三层统一模型¶

    ┌──────────────────────────────────────────────────┐
│ 第3层: 信号槽（解耦通知）                          │
│   signal → connect → slot                         │
│   决定"谁通知谁"，连接类型决定执行方式              │
├──────────────────────────────────────────────────┤
│ 第2层: 事件系统（消息分发）                        │
│   QEvent → event() → specific handler             │
│   QueuedConnection 将槽调用封装为 QMetaCallEvent   │
├──────────────────────────────────────────────────┤
│ 第1层: 事件循环（驱动引擎）                        │
│   QEventLoop → processEvents() → 取事件 → 分发     │
│   UI刷新、定时器、网络回调、排队槽都依赖它          │
└──────────────────────────────────────────────────┘

    关键洞察：排队信号槽（QueuedConnection）在第3层看起来是"信号触发了槽"，但在第2层实际上是"信号发射时创建了一个 QMetaCallEvent，投递到接收者线程的事件队列"。接收者的第1层事件循环取出这个事件并执行槽函数。三层的协同决定了槽函数在哪个线程、什么时机执行。

    2. 连接类型 = 三层协同的调度策略¶

    连接类型 信号线程 = 接收者线程？ 执行方式 典型场景
    DirectConnection 不关心 信号发射时直接调用槽函数（同调用栈） 同线程对象间通知
    QueuedConnection 不关心 封装为事件 → 投递到接收者线程事件队列 → 事件循环中执行 跨线程通信的标准方式
    AutoConnection（默认） 同线程 退化为 DirectConnection 最常用
    AutoConnection（默认） 不同线程 退化为 QueuedConnection 跨线程安全通信
    BlockingQueuedConnection 不同线程 QueuedConnection + 信号线程阻塞等待槽执行完毕 需要返回值的跨线程调用

    postEvent vs sendEvent¶

    // sendEvent: 同步——立即调用接收者的 event()，等待处理完毕才返回
QCoreApplication::sendEvent(receiver, event);

// postEvent: 异步——将事件加入接收者线程的事件队列，立即返回
QCoreApplication::postEvent(receiver, event);

    面试关键：postEvent = 排队投递（类似 QueuedConnection），sendEvent = 直接调用（类似 DirectConnection）。混用二者会导致槽函数执行线程与预期不符。

    3. 线程亲和性 = 调度目的地¶

    每个 QObject 有线程亲和性（thread affinity）——它"属于"哪个线程：

    • 创建时：亲和性 = 当前线程
    • moveToThread()：改变亲和性到目标线程
    • 影响：QueuedConnection 的槽函数在亲和性线程执行；定时器、网络回调需要亲和性线程有事件循环
    // 经典模式：工作对象移动到子线程
QThread* thread = new QThread;
Worker* worker = new Worker;   // 先创建在主线程
worker->moveToThread(thread);  // 亲和性改为子线程

// 跨线程连接：信号在子线程发射，槽在主线程执行
connect(worker, &Worker::DataReady,
        widget, &Widget::OnDataReady);  // AutoConnection → 跨线程 → Queued

thread->start();

    子线程为什么必须有事件循环？¶

    QueuedConnection 把槽调用封装为事件投递到目标线程。如果目标线程没有事件循环（没调用 exec()），事件永远留在队列中——槽不会被执行。子线程要处理排队槽、定时器、网络事件，必须运行事件循环：

    // 子线程的 run() 或直接在子线程中
QThread::exec();  // 启动事件循环
// 或者等效的局部事件循环

    4. 常见面试交叉点¶

    面试问题 涉及的三层
    "信号槽跨线程时槽在哪个线程执行？" 第3层（连接类型）+ 第1层（事件循环）+ 亲和性
    "为什么子线程直接 setText 会崩溃？" 亲和性 + 事件系统：UI 对象属于 GUI 线程
    "排队的槽为什么不执行？" 目标线程可能没有事件循环
    "postEvent 和 sendEvent 区别？" 第2层事件系统的同步/异步分发
    "BlockingQueuedConnection 会死锁吗？" 如果接收线程也在等发送线程——形成等待环

    5. 关联概念¶

    • moc — MOC 生成的代码是信号槽分发的编译期基础
    • qt-plugin-system — 插件内的 QObject 生命周期也受线程亲和性约束
    • condition-variable — 条件变量是底层同步原语，Qt 在此之上构建了事件驱动的异步模型
    • thread-pool — 线程池 + QObject::moveToThread() 是 Qt 中处理并发任务的两种模式
